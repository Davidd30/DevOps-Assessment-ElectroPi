# DevOps-Assessment-ElectroPi

A simple three-tier app (frontend, backend API, database) deployed on AWS using Terraform, Docker, and GitHub Actions.
 
## Architecture
 
VPC with public and private subnets. Backend runs as a Flask API on ECS Fargate, behind an ALB. Database is RDS Postgres, private subnet only. Frontend is a static site on S3. DB credentials live in Secrets Manager, not in code. Logs and one alarm go to CloudWatch.
 
See `docs/architecture.png` for the diagram.
 
## Why these choices
 
- **AWS** — most familiar to me, easy free tier setup.
- **ECS Fargate** — no servers to patch, and it covers both "compute" and "containerization" in one piece.
- **Terraform, split into modules** (`network`, `database`, `compute`) — easier to read and reuse.
- **RDS in private subnets only** — no public access to the database.
- **Secrets Manager** — DB password is generated and stored there, never typed in a file.
- **GitHub Actions with OIDC** — no AWS access keys stored anywhere. GitHub asks AWS for a short-lived token at run time.
## Run locally
 
```bash
cd app
pip install -r requirements.txt
python backend.py
```
 
Or with Docker:
 
```bash
docker build -f app/Dockerfile -t myapp-backend .
docker run -p 5001:5001 myapp-backend
curl http://localhost:5001/health
```
 
## Deploy to AWS
 
```bash
cd infra
terraform init
terraform apply -var="container_image=<ecr-url>:<tag>"
```
 
State is stored remotely in an S3 bucket, with locking. The bucket was created once by hand before the first `terraform init` (see `backend.tf`).
 
## CI/CD pipeline
 
`.github/workflows/deploy.yml` runs on every push to `main`:
 
1. Checkout code
2. Install deps, run tests (pytest)
3. Auth to AWS (OIDC, no stored keys)
4. Build Docker image
5. Scan image with Trivy
6. Push image to ECR
7. `terraform apply` to update ECS with the new image
**Secrets**: AWS access uses OIDC federation, not stored keys. DB credentials are generated by Terraform and stored in Secrets Manager, and only referenced by ARN in the ECS task definition.

## Trade-offs made because of the time limit
 
- No NAT Gateway, so private subnets have no outbound internet access.
- IAM role used by GitHub Actions has `AdministratorAccess` — too broad for production, scoped-down policy would be used instead.
- RDS is single-AZ, not multi-AZ.
- CloudWatch alarm has no email/SNS target attached yet.
- Trivy scan reports vulnerabilities but does not block the pipeline (`exit-code: 0`).
- Local dev used Python 3.14, container uses Python 3.13 — small version gap.
## If this were a real production system
 
**Scale**: turn on ECS service auto scaling based on CPU/request count. Add an RDS read replica if reads grow.
 
**Cost**: Fargate is convenient but costs more per vCPU than EC2 at steady, high load — would compare against EC2 with reserved capacity once traffic is predictable. Reserved RDS instances would cut database cost too.
 
**High availability**: turn on Multi-AZ for RDS, run at least 2 ECS tasks across both AZs, add a NAT Gateway per AZ so private subnets keep working even if traffic needs outbound internet.
